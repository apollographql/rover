---
title: Validating Client Operations
subtitle: Confirm that all client operations are supported by your schema
---

You can confirm that all of the GraphQL operations defined in your client application are valid against the types, fields, and arguments in your graph's published schema. To do so, use the [`rover client check`](./commands/client#client-check) command:

```bash
rover client check my-graph@my-variant --include "src/**/*.graphql"
```

The argument `my-graph@my-variant` is a [graph ref](./conventions/#graph-refs) that identifies the GraphOS graph and [variant](/graphos/graphs/#variants) to validate against. If you omit the variant, Rover uses the default variant named `current`.

For each operation, `rover client check` reports whether it passed validation and, for failures, the specific error:

```
Validated 4 operations (3 files scanned)
GetTeamGrid: src/components/TeamGrid.graphql:4:1
  FAILURE Field "league" must not have a selection since type "SportsLeague" has no subfields.

4 total operations validated
1 failure
```

The command exits with a non-zero exit code if any operation fails validation.

## Scanning operations

Run this command from your client application's root directory. Rover scans for `.graphql` files using the glob patterns you provide with `--include`. You can specify multiple patterns and optionally exclude paths with `--exclude`:

```bash
rover client check my-graph@staging \
  --include "src/**/*.graphql" \
  --exclude "src/__generated__/**"
```

If your operations are embedded in TypeScript source files as `gql` tagged template literals, use [`rover client extract`](./commands/client#client-extract) first to generate standalone `.graphql` files, then run `rover client check` against those.

```bash
rover client extract --out-dir graphql
rover client check my-graph@staging --include "graphql/**/*.graphql"
```

All operations must have names. Anonymous operations are rejected with an error.

## Authentication

Make sure you set the `APOLLO_KEY` environment variable so Rover can authenticate with GraphOS and identify which graph's schema to check against:

```bash
APOLLO_KEY=your-api-key rover client check my-graph@my-variant \
  --include "src/**/*.graphql"
```

## Checking against a particular variant

During development, your application might be executing GraphQL operations against a locally running server with a schema that differs from your production server's schema. Because schemas can differ, it's important to always check your operations against the schema for a particular environment before you push client changes to that environment.

Specify the target variant, `production` in this example, directly in the graph ref:

```bash
rover client check my-graph@production --include "src/**/*.graphql"
```

## Using with continuous integration

Your application can incorporate `rover client check` into its continuous delivery pipeline to help ensure that new and modified operations are valid before they go live.

For example, the following GitHub Actions configuration excerpt incorporates the command:

```yaml title=".github/workflows/check.yml"
- name: Validate client operations
  run: |
    rover client check my-graph@production \
      --include "src/**/*.graphql"
  env:
    APOLLO_KEY: ${{ secrets.APOLLO_KEY }}
```
